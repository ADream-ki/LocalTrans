# LocalTrans 双向实时翻译方案对比

## 1. 候选方案
- 方案 A: 维持现状（双 `RealtimePipeline` 并行，轻量增强）
- 方案 B: 会话编排层（`Session Orchestrator + Worker`，推荐）
- 方案 C: 单流水线多角色（统一流 + 说话人分离/角色路由）

## 2. 对比表

| 维度 | 方案 A: 双 Pipeline | 方案 B: Orchestrator + Worker | 方案 C: 单流水线多角色 |
|---|---|---|---|
| 架构复杂度 | 低 | 中 | 高 |
| 开发成本 | 低 | 中 | 高 |
| 双向稳定性 | 中 | 高 | 中 |
| 时延可控性 | 中 | 高 | 中 |
| 资源占用 | 中-高 | 中（可优化复用） | 低-中 |
| 故障隔离 | 高 | 高 | 低-中 |
| 可观测性 | 中 | 高 | 中 |
| 多方会话扩展 | 低 | 高 | 中 |
| 回滚难度 | 低 | 中 | 高 |
| 推荐等级 | 可继续使用 | 推荐下一阶段 | 暂不建议当前阶段 |

## 3. 结论
- 在你当前代码基础上，`方案 B` 是更优解:
  - 能保持现有双向逻辑稳定；
  - 可在不重写 ASR/MT/TTS 的前提下提升可观测性和可维护性；
  - 为后续多人会议和策略引擎预留扩展位。

## 3.1 当前实现进度（截至当前版本）
- 已完成:
  - `SessionOrchestrator` 编排壳层（CLI/GUI 双向会话已接入）
  - 跨方向防回授抑制（减少回声引起的自激循环）
  - 方向级自动恢复（单方向异常时自动重启）
- 未完成:
  - Worker 接口彻底抽象
  - 共享上下文（术语/缓存/策略）统一模块化
  - 策略引擎（动态调参与自适应降级）

## 4. 为什么不优先方案 C
- 单流水线要依赖更强的说话人分离与角色识别，当前本地实时链路中会引入更高误判风险。
- 一旦角色路由错误，会同时污染双向结果，排障成本显著高于双 Worker 隔离架构。

## 5. 落地建议（基于当前仓库）
- 短期（1-2 周）:
  - 巩固 `SessionOrchestrator`（方向健康状态、错误码分类、恢复策略参数化）。
  - 将 GUI/CLI 会话摘要与日志字段统一（便于跨端排障）。
- 中期（2-4 周）:
  - 抽象 Worker 接口，把当前 `RealtimePipeline` 作为默认实现。
  - 接入统一指标: 每方向 ASR/MT/TTS 延迟、丢帧率、重启次数。
- 长期（4+ 周）:
  - 策略引擎（动态 buffer、自动降级）与多方会话路由。

## 6. 决策建议
- 如果你当前目标是“尽快稳定可用”: 先做 `方案 A+`（补监控和回退开关）。
- 如果你目标是“可持续扩展 + 团队协作”: 直接进入 `方案 B` 的增量改造。
